🤖 AI-Enhanced Classification

Large Language Models (LLMs) can significantly improve classification accuracy for ambiguous cases, provide natural language explanations, and handle complex supplier descriptions. This page outlines integration strategies using Claude, GPT-4, or open-source models.

When to Use LLMs

LLMs should augment, not replace, the rule-based classification engine. Best used for:

📝 Ambiguous Cases (20-70% Confidence)

When rule-based classifier is uncertain, LLM can analyze full context (supplier name, commodity, description, historical patterns) to make informed decision.

Example: "ALPHA TECH SOLUTIONS" with commodity "IT Consulting" could be Development, Security, or Infrastructure. LLM reads description to determine correct classification.

🆕 New Suppliers (0-20% Confidence)

First-time suppliers with no historical data. LLM can infer activity code from business description and commodity type.

Example: "BETA CLOUD ANALYTICS INC" - no history. LLM recognizes "Cloud Analytics" → suggests PL210 (Data Analytics) or IN100 (Compute).

💬 Natural Language Explanations

Generate human-readable rationale for each classification decision to help users understand why a code was assigned.

Example: "Assigned DL320 (Cyber Security) because supplier 'Fortinet' is a known security vendor and commodity 'Firewall Management' matches cyber security keywords."

🔍 Complex Multi-Product Suppliers

Suppliers selling diverse products (CDW, Amazon) where description analysis is critical. LLM excels at understanding context.

Example: "CDW - Microsoft Office 365 Licenses" → WP210 (Collaboration) vs. "CDW - Dell Laptops" → WP100 (Client Computing).

📚 Learning from Corrections

Analyze patterns in manual corrections to suggest new classification rules or keyword mappings.

Example: LLM notices "VPN" frequently corrected to WP320 → suggests adding "VPN" as WP320 keyword.

🌐 Multi-Language Support

Classify invoices with non-English supplier names or descriptions by translating and understanding context.

Example: "SOCIÉTÉ DE SERVICES INFORMATIQUES" → LLM recognizes French for "IT Services" → suggests DL100.

Integration Architecture

Implementation Options

Option 1: Anthropic Claude API (Recommended)

# classification/llm/claude_classifier.py import anthropic from typing import Dict, List class ClaudeClassifier: def __init__(self, api_key: str): self.client = anthropic.Anthropic(api_key=api_key) def classify_with_llm( self, invoice_line: Dict, rule_based_suggestions: List[Dict], activity_codes: List[Dict] ) -> Dict: """Use Claude to classify ambiguous invoice.""" prompt = f"""You are an expert at classifying procurement invoices into activity codes. Invoice Details: - Supplier: {invoice_line['supplier']} - Commodity: {invoice_line['commodity']} - Description: {invoice_line['description']} - Amount: ${invoice_line['amount']} Rule-based classifier suggestions (low confidence): {self._format_suggestions(rule_based_suggestions)} Available Activity Codes: {self._format_activity_codes(activity_codes)} Task: Determine the most appropriate activity code for this invoice line. Respond in JSON format: {{ "activity_code": "CODE", "confidence": 0-100, "rationale": "Clear explanation of why this code is correct", "alternative_codes": ["CODE1", "CODE2"] }}""" message = self.client.messages.create( model="claude-3-5-sonnet-20241022", max_tokens=1024, messages=[{"role": "user", "content": prompt}] ) import json result = json.loads(message.content[0].text) # Add method tracking result["method"] = "claude_llm" result["model"] = "claude-3-5-sonnet" return result def _format_suggestions(self, suggestions: List[Dict]) -> str: return "\n".join([ f"- {s['code']} ({s['confidence']}%): {s['rationale']}" for s in suggestions ]) def _format_activity_codes(self, codes: List[Dict]) -> str: # Summarize codes to fit context window return "\n".join([ f"{c['code']}: {c['level3_name']} - {c['definition'][:100]}" for c in codes ])

✓ Advantages

  • Best-in-class reasoning for complex cases
  • Extended context window (200K tokens)
  • Structured JSON output
  • Fast response times (2-3 seconds)
  • Reliable, production-ready API

✗ Considerations

  • Cost: $3/million input tokens, $15/million output
  • Requires Anthropic API key
  • External API dependency
  • Rate limits apply

Option 2: OpenAI GPT-4

# Alternative: OpenAI GPT-4 import openai def classify_with_gpt4(invoice_line, suggestions, activity_codes): response = openai.ChatCompletion.create( model="gpt-4-turbo-preview", messages=[ {"role": "system", "content": "You are an expert at classifying procurement invoices."}, {"role": "user", "content": prompt} ], response_format={"type": "json_object"} ) return json.loads(response.choices[0].message.content)

Option 3: Local Open-Source Model (Llama 3, Mixtral)

# Local deployment using Ollama or vLLM import requests def classify_with_local_llm(invoice_line, suggestions): # Assumes Ollama running locally response = requests.post( "http://localhost:11434/api/generate", json={ "model": "llama3:70b", "prompt": prompt, "format": "json" } ) return response.json()

✓ Advantages

  • No per-request costs (after hardware)
  • Data stays on-premises (privacy)
  • No rate limits or API dependencies
  • Full control over model and tuning

✗ Considerations

  • Requires GPU infrastructure (~$5K-20K)
  • DevOps overhead (deployment, monitoring)
  • Lower quality than Claude/GPT-4
  • Slower inference (5-10 seconds)

Prompt Engineering Best Practices

Effective Prompt Structure

You are an expert at classifying procurement invoices into activity codes. ### Context Our company uses 59 activity codes across 5 categories: - Business (BS): Revenue operations, CRM, sales - Delivery (DL): Development, operations, security - Infrastructure (IN): Compute, data center, network - Platform (PL): Applications, data, development platforms - Support/Corporate (SC): Accounting, legal, HR, facilities - Workplace (WP): Client devices, collaboration, productivity ### Invoice Details Supplier: {{supplier}} Commodity: {{commodity}} Description: {{description}} Amount: ${{amount}} ### Historical Data (if available) This supplier has been classified as: - {{past_code_1}} (3 times) - {{past_code_2}} (1 time) ### Rule-Based Suggestions (Low Confidence) 1. {{code_1}} ({{conf_1}}%): {{rationale_1}} 2. {{code_2}} ({{conf_2}}%): {{rationale_2}} 3. {{code_3}} ({{conf_3}}%): {{rationale_3}} ### Available Activity Codes {{full_list_of_codes_with_definitions}} ### Task Determine the most appropriate activity code. Consider: 1. Supplier business type (what do they sell?) 2. Commodity code semantics 3. Description details 4. Historical patterns 5. Consistency with similar invoices Respond in JSON: { "activity_code": "CODE", "confidence": 0-100, "rationale": "Why this code? Reference specific details.", "alternative_codes": ["CODE1", "CODE2"], "flags": ["non_it_vendor", "ambiguous", "high_value"] }

Tips for Better Results

  • Include Definitions: Provide activity code definitions, not just codes
  • Show Examples: Include 2-3 past classifications for similar suppliers
  • Request Reasoning: Ask for rationale to validate LLM logic
  • Structured Output: Use JSON mode for reliable parsing
  • Add Context: Historical data improves accuracy significantly
  • Temperature: Use low temperature (0.2-0.3) for consistent classifications

Implementation Roadmap

Phase 1: Proof of Concept (Week 1-2)

  • Integrate Claude API with on-demand "AI Suggest" button
  • Test with 50 low-confidence invoices from SAMPLE.xlsx
  • Measure accuracy improvement vs. rule-based alone
  • Estimate cost per invoice

Phase 2: Selective Automation (Week 3-4)

  • Auto-call LLM for 40-70% confidence range
  • If LLM confidence ≥80%, auto-assign
  • If LLM confidence 50-80%, add to suggestions
  • Monitor cost and accuracy metrics

Phase 3: Optimization (Month 2)

  • Fine-tune confidence thresholds
  • Implement prompt caching to reduce costs
  • Add batch processing for overnight runs
  • Consider local LLM for cost savings (if volume high)

Phase 4: Advanced Features (Month 3+)

  • LLM analyzes manual corrections to suggest rule improvements
  • Natural language query interface ("Show me all security spend")
  • Anomaly detection (unusual supplier/activity combinations)
  • Multi-language invoice support

Recommended Approach

Start Rule-Based Only (Phase 1 of main project): Build core classification engine without LLM. Measure baseline accuracy and auto-assignment rate.

Add LLM Selectively (Phase 2, optional): After validating rule-based approach, add LLM for ambiguous cases (40-70% confidence). Start with on-demand user-triggered, expand to automatic based on results.

Total Cost: Rule-based system: $30K dev + $100/month infra. + LLM: +$1-2K dev + $30-90/month API costs.
